# Pythonスクレイピングのベストプラクティス2026 — 法律・技術・運用の最新事情

Webスクレイピングは2026年現在も、データ収集の主要な手段であり続けています。一方で、サイト側の対策（Cloudflare、bot検知）の高度化、各国の法規制の整備、そしてAI訓練データに対する社会的な議論が進み、「技術だけ知っていれば良い」時代は完全に終わりました。本記事では、Pythonでスクレイピングを実装・運用する人向けに、技術・法律・運用の3つの側面から最新のベストプラクティスを整理します。

## 法律と利用規約 — まず確認すべきこと

### 日本の法的整理

日本でのスクレイピングは、原則として違法ではありません。著作権法第30条の4により、**情報解析目的**での複製は適法とされています。ただし次の3点には注意が必要です。

**サーバへの過剰負荷** — 短時間に大量のリクエストを送る行為は、業務妨害罪に問われる可能性があります。実際に2010年の岡崎市立図書館事件では、1秒に1リクエスト程度でも逮捕者が出ています（後に不起訴）。安全圏は1リクエスト/秒以下、夜間に集中させない、などの配慮が必須です。

**ログイン後コンテンツ** — 利用規約で明示的にスクレイピングを禁じているサイト（Twitter/X、LinkedIn、Amazonなど）にログインして取得する行為は、契約違反として民事責任を問われる可能性があります。

**個人情報の収集** — 個人情報保護法の対象となります。氏名・連絡先などを集める場合は利用目的の通知や保管ルールが必要です。

### 海外サイトを対象にする場合

EUのGDPR、米国カリフォルニア州のCCPAなど、対象国の法令も適用されます。特にEUは個人情報の取り扱いに厳しく、違反時の制裁金が高額（最大2,000万ユーロまたは年間売上の4%）です。海外サイトを対象にする場合は、事前に対象国の法令を確認するか、専門家への相談を強く推奨します。

### robots.txtと利用規約の確認手順

実装前に必ず次の3つをチェックします。

1. `https://対象サイト/robots.txt` を確認
2. サイトの利用規約（Terms of Service）でスクレイピング禁止条項を検索
3. APIが提供されているなら、まずそちらを使えないか検討

robots.txtに法的拘束力はありませんが、サイト運営者の意思表示として尊重するのが業界慣行です。Crawl-delayの指定があればそれに従いましょう。

## 技術選択 — ライブラリの使い分け

### 静的サイトには requests + BeautifulSoup

JavaScriptでの動的レンダリングがない普通のHTMLサイトには、`requests` + `BeautifulSoup4` の組み合わせが鉄板です。軽量で高速、コードも数行で書けます。

```python
import requests
from bs4 import BeautifulSoup
import time

session = requests.Session()
session.headers.update({"User-Agent": "MyResearchBot/1.0 (contact@example.com)"})

resp = session.get("https://example.com/list", timeout=10)
soup = BeautifulSoup(resp.text, "html.parser")
items = [a.text.strip() for a in soup.select(".item-title")]
time.sleep(1)  # 必ず間隔を空ける
```

User-Agentに連絡先を含める習慣をつけると、サイト運営者からの問い合わせがメールで来るようになり、トラブル回避になります。

### 動的サイトには Playwright

JavaScriptで内容が生成されるSPAサイト（React/Next.js/Vueなど）には、ヘッドレスブラウザを使う必要があります。2026年時点での第一選択は **Playwright** です。Seleniumより高速で、API設計が現代的、Microsoftが活発に開発しています。

```python
from playwright.sync_api import sync_playwright

with sync_playwright() as p:
    browser = p.chromium.launch(headless=True)
    page = browser.new_page()
    page.goto("https://example.com/dashboard")
    page.wait_for_selector(".loaded")
    titles = page.locator(".card-title").all_inner_texts()
    browser.close()
```

`wait_for_selector` で要素の出現を待つこと、明示的なタイムアウト設定をすること、これらが安定運用のコツです。

### スケール時には Scrapy

数千ページ以上を扱うなら、フレームワーク **Scrapy** の出番です。並列実行・自動リトライ・パイプライン処理が組み込まれており、結果のJSONL/CSV出力、Cloudベースの分散実行（Scrapy Cloud）まで一気通貫で揃います。学習コストは少し高いですが、本格運用には不可欠です。

## 運用設計 — 安定して回し続けるために

### ブロック対策とその限界

サイト側の検知技術は急速に進化しています。**過度な対策はそれ自体が攻撃的とみなされる**可能性がある点に注意してください。やるべきことは次のレベルまでです。

- User-Agentを実在のブラウザに合わせる
- リクエスト間隔をランダム化（1-3秒）
- 同一IPからの連続リクエストを避ける（プロキシ利用は規約と相談）
- セッションCookieを保持する

逆に、CAPTCHA自動突破ツールや、検知回避を売りにする商用プロキシの利用は法的・倫理的グレーゾーンです。仕事で使うなら避けた方が無難です。

### エラーハンドリングと再開可能性

長時間動かすスクリプトでは、途中で落ちたときの再開可能性が重要です。最低限、次の3つを実装しましょう。

1. **チェックポイント保存** — 取得済みURLを定期的にファイルに記録
2. **指数バックオフ** — 連続失敗時は2秒→4秒→8秒と待ち時間を増やす
3. **構造化ログ** — JSONL形式で記録すると後の分析が楽

`tenacity` ライブラリを使えばリトライ処理は数行で書けます。

### データの保存と更新管理

取得したデータの保存先は、用途に応じて選びます。

| 用途 | 推奨 |
|---|---|
| 数百〜数万件の一回限り | CSV / JSONL |
| 継続的な蓄積・差分管理 | SQLite / PostgreSQL |
| 大規模・複数人で共有 | BigQuery / S3 + DuckDB |

**差分更新の仕組み**を最初から組み込んでおくと、運用コストが激減します。前回取得時のハッシュ値や更新日時を保存し、変更があったレコードだけ処理する設計です。
